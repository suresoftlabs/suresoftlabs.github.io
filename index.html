CODESCROLL 의 철 지난(?) 기술

CODESCROLL 개발실

안녕하세요, CODESCROLL 개발실내에 엔진기술을 담당하고 있는 권민혁입니다(회사에서 쓰는 자유로운 블로그라 기분이 좀 묘하네요^^)

우리는 CODESCROLL 이라는 브랜드 아래에 세가지 제품을 개발하고 있습니다. 코드를 분석해서 오류를 잡아주는 Code Inspector, SNIPER 와 유닛테스트를 할 수 있는 Controller Tester 가 있죠. 일반 개발자가 구입할 수 있는 가격대는 아니고, S/W 품질을 중요하게 생각하는 기업이 저희 개발실의 주 고객입니다. 이런 기업들이 만든 S/W 는 S/W 의 동작 실패가 인명이나 재산상의 중대한 손실을 초래하게 됩니다.

CODESCROLL 개발실은 현재(2017년 11월) 15명이 근무하고 있습니다.

네!, 세가지 제품을 유지보수하기에는 턱없이 부족한 숫자죠. 그렇지만 저희는 그 어려운 일을 하고 있습니다^^ 어떻게? 더 효율적으로 일하고, 문제를 최대한 고립하고 조금씩 개발하면서 말이죠. 그렇다고 경쟁사보다 느리게 개발하거나 품질이 떨어지게 개발하진 않습니다. 적당한 균형점을 찾아서 개발하는 중입니다. 효율적이지 않으면 우리가 삶을 온전히 누리기 힘들기 때문에 지금도 계속해서 더 효율적으로 일하는 방법이 무엇인지 고민하고 있습니다. (feat. 놀기도 해야하고 연애도 해야하고 육아를 해야 해요…)

개발실은 업무를 진행하면서 업무 효율을 돕는 여러 시스템을 사용합니다. 업무 효율을 돕는 여러가지 방법에 대해서는 나중에 이야기 할 기회가 있을 것 같습니다.

이제 본격적으로 CODESCROLL 개발실의 철지난(?) 기술에 대해 이야기 해보죠. 설명할 기술은 저희가 다루는 기술의 10 분의 1 정도 일겁니다. 그냥 제가 제일 관심있는 부분만 한번 보죠.

Inside CODESCROLL(분석엔진 편)

Source Code Parser

CODESCROLL 내의 핵심 기술은 단연 제가 만들고 있는 프로그램 분석 엔진입니다(UI 는 다음에…).

프로그램 분석 엔진은 쉽게 말하면 컴파일러라고 생각하면 됩니다. 오브젝트 코드를 생성하지 않는 대신, 저희만의 파일 포맷으로 소스코드의 구조정보등을 출력합니다.
이러한 코드의 구조정보는 나중에 UI 에서 다양하게 표현되죠. 이를 테면, Control Flow Graph 를 표시한다거나, 함수간의 호출 관계를 표시한다거나 말입니다.

프로그램 분석 엔진을 만드는 일은 생각보다 어렵지 않습니다. 적당한 파서를 이용해서 파싱된 트리를 순회하는 로직을 작성해주면 되죠. 이건 기술이 아닙니다. 누구나 만들 수 있으니깐요.

진짜 기술은 160개가 넘는 서로 다른 컴파일러를 지원하는 것에 있습니다. 이게 무슨 말이냐면, C/C++ 코드는 국제 표준에서 많은 부분을 Implementation-defined(만드는 사람이 알아서 정함) 으로 지정하고 있습니다. 즉, 같은 코드라도 컴파일러마다 조금씩 다르다는 겁니다. 언어의 표현력도 다르고요. 이런 것들을 하나의 AST 로 표현해 내는 것도 어렵지만, 파서를 자유롭게 옵션처리(?) 하는 것도 매우 어렵습니다. 거기에 새로운 컴파일러가 주어졌을때, 이 녀석이 어떤것들을 지원하고 지원하지 못하는지를 알아내는 과정도 그리 순탄치 않습니다.

Code Error Detection

또 다른 기술은 파스 트리만 보고 오류를 검출하는 기술입니다. 관상쟁이가 얼굴만 보고 팔자가 어떨지 보는 것과 비슷합니다. 그렇다고 저희가 사용하는 기술이 대충 때려 맞추는 기술은 아니고, 수학적인 엄밀함을 가지고 있지요~ 이것도 나중에 다시 설명하기로 해요(복잡). 어쨋건, 우리 코드 분석 엔진은 코드를 보고 어떤 변수가 어떤 값을 가질지를 알 수 있습니다. 어떤 포인터가 언제 생성되는지 해제되는지도 알 수 있고요. 이런 정보를 잘 조합하면, 프로그램 작동중에 발생할 문제들을 점쟁이처럼 ‘척’ 하고 알아 낼 수 있습니다. 신기하죠?

Code Transformation

또 있습니다^^ 우리 코드 분석 엔진의 주요 기능으로 코드를 원하는 모양으로 바꿀 수 있는 구조가 있거든요? 우리끼리는 CodeRewriter 라고 합니다. 이 녀석은 코드 사이사이에 새로운 코드를 넣을 수도 있고, 일부 코드는 삭제할 수도 있고, 다른 곳으로 이동시킬 수 있는 녀석입니다. 네!, 맞습니다. 리팩토링 도구를 만들 수 있는 기능이죠. Transformation 만 주로 수행합니다. 프로그램의 의미 따위는 아예 관심이 없죠.

Code Conversion

앞서 언급한 CodeRewriter 가 있기 때문에, 코드를 마음대로 주무룰수 있습니다. 여기에 의미있는 해석을 덧붙이면 좋은 응용프로그램을 만들 수 있죠. 즉, Linux 에서 작동하는 프로그램은 Windows 에서 작동하는 프로그램으로 개조할 수 있는 겁니다. 플랫폼에 의존적인 코드들을 싹~ 드러내고, Windows 에 의존하게 바꿀 수 있는 거죠. 그리고 나서 Windows 용 컴파일러로 컴파일하면 되니깐요. 지금은 Code Converter 를 우리 도구를 위해서만 사용하고 있지만, 언젠가 멀티플랫폼을 위한 코드 관리 도구로서의 기능을 갖추고 일반에게 공개할 수 있었으면 합니다.

그 밖에

코드 패턴 매칭 기술, SAT Solver, Symbolic Execution 등 CODESCROLL 에 녹아있는 기술들은 많습니다. 컴퓨터 과학의 가장 기반에 있는 기술이라 잘 변하지 않는 것 뿐이지 더디긴 해도 조금씩 변하고 있긴 합니다. SAT Solver 대신 SMT Solver 를 쓰는 엔진이 늘어나고 있고, Abstract Intepretation 대신 Symbolic Execution 을 기반으로 오류를 검출하는 엔진도 늘어나고 있습니다.

맺은말

내용을 자세히 설명하지 않아도 철 지난 기술들입니다. 소스코드를 파싱하는 방법은 이미 70~80년대에 완성된 기술이고, 코드 이동이나 코드 변환도 마찬가지 입니다. 하지만 엔지니어들에 의해 잘 구현되지 않는 기술이기도 하죠. 어느 누가 컴파일러를 만들려고 하겠습니까? 그런데 말입니다~ 이 기술들이 이제는 팔 수 있을 정도가 된것 같습니다. 그래서 더 기대되기도 합니다. 혹시 Clang 에서 제공하는 라이브러리 모음으로 무언가를 만든 경험이 있나요? 그렇다면 저희팀에 딱 맞네요^^ 포지션은 언제든지 열려있습니다. 다만 지원자가 없을 뿐…(AST를 이해하는)

오늘은 이정도까지만 하지만, 다음에는 조금 더 깊이 있는 주제를 이야기 했으면 좋겠습니다. 또 뵈요~ 여러분~

기술 블로그를 시작합니다.

무슨 이야기를 하나요?

대부분의 기술 블로그가 그렇듯 우리도 슈어소프트 연구소에서 일어나는 재밌는 일들, 그리고 우리가 매일 다루는 다양한 문제 유형들에 대해서 작성해 볼까합니다.

그리고 가끔, 고객에게 가치를 전달하기 위해 벌어지는 스펙터클한 전투현장에 대한 이야기도 다룰 수 있었으면 좋겠습니다.

많은 시간이 지나서 우리가 스스로를 돌아볼 때, 이것이 슈어소프트 연구소의 문화였음을 기억하고 더 발전적인 방향으로 이끄는 자가 되기 위해 이 블로그를 시작합니다.
